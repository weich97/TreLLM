# ICLR Submission Roadmap

This roadmap turns TreLLM's research direction into a set of submission-grade
readiness gates. It is intentionally written as an engineering contract:
each gate has a claim, the evidence needed to support it, and the repository
work that must exist before the claim belongs in a main-conference paper.

## Target Positioning

TreLLM should be presented as an LLM financial-agent audit and reliability
system. TradeArena should be presented as the public leaderboard layer for
ranking comparable, auditable agent runs.

The paper should not claim that LLMs can trade profitably. The stronger claim is
that LLM financial-agent evaluations are unstable unless execution assumptions,
risk intervention, memory contamination, auditability, and provider drift are
treated as first-class protocol variables.

Suggested thesis:

> LLM financial-agent evaluation is not stable under execution assumptions,
> memory contamination, and audit-trace defects. TreLLM provides a replayable
> protocol that exposes these failure modes and prevents leaderboard results
> from being over-read as trading-profit claims.

## Recommended Paper Shape

The most defensible ICLR route is an agent evaluation and reliability paper:

1. Define TreLLM as the audit/control system and TradeArena as the public
   leaderboard surface.
2. Show that execution assumptions can change measured outcomes and model
   rankings.
3. Show that contaminated or misleading memory can amplify unsafe financial
   decisions.
4. Show that LLMs can be evaluated as auditors of financial traces using
   injected defects with machine-checkable ground truth.
5. Release the protocol, manifests, statistics, and reproduction artifacts so
   outside users can verify the claims without private raw prompts or accounts.

The negative result is allowed to be central: if realistic execution and risk
controls erase apparent LLM trading skill, that is a useful reliability finding.

## Readiness Gates

### G0: Scope and Identity

Claim:
TreLLM is a system for auditable LLM financial-agent decisions; TradeArena is
the public leaderboard for comparable runs.

Required evidence:
- README and launch metadata use TreLLM for the system name.
- TradeArena wording is limited to leaderboard, ranking, benchmark-card, or
  public submission surfaces.
- Public pages avoid implying that the repository is only a trading leaderboard.

Repository work:
- Keep `scripts/check_release_readiness.py` guarding public identity phrases.
- Keep GitHub About metadata aligned with `docs/launch/README.md`.

### G1: Direct API Provider Protocol

Claim:
Main paper results are produced through versioned direct provider APIs, not
through provider-routing surfaces whose model identity or prompt wrappers cannot
be controlled.

Required evidence:
- A direct API runner records `provider`, `model_id`, version or release label,
  temperature, sampling parameters, call window, prompt hash, response hash, and
  redacted manifest hash.
- Poe or other routed-provider rows are separated as historical or appendix
  evidence and are not used for headline model claims.
- Failed, retried, cached, and partially parsed calls have explicit accounting.

Repository work:
- Add or harden the direct API runner.
- Add provider manifest schema checks.
- Add a result-table field that distinguishes `direct-api` from `routed-provider`
  and `deterministic-baseline`.
- Keep the v0.3 direct API matrix gate generated by
  `scripts/build_v03_direct_api_matrix_gate.py`; it verifies manifest hash
  bindings and labels rows below 10 seeds and 3 samples per seed as
  pilot/incomplete evidence.
- Keep the v0.3 direct API matrix plan generated by
  `scripts/build_v03_direct_api_matrix_plan.py`; it pre-registers provider,
  scenario, seed, sample, contamination-tier, and execution-level rows and
  records credential environment-variable presence without making live calls.
- Keep the no-key direct API call-packet artifact generated by
  `scripts/build_v03_direct_api_call_packets.py`; it turns pre-registered rows
  into hash-bound execution packets, expected manifest paths, expected
  submission paths, and redaction contracts without publishing raw
  prompts/responses.
- Do not close the direct-model evidence gap until non-fixture direct API rows
  pass both the preflight plan and the matrix gate.

### G2: Execution Assumption Ladder

Claim:
Model rankings and risk conclusions must be reported with execution assumptions.

Required evidence:
- E0 ideal execution, E1 paper stress, E2 parameter sweep, and E3 quote/fill or
  calibrated replay are defined in a frozen protocol.
- Each reported model has paired runs across the ladder where feasible.
- Ranking stability is reported with Kendall tau, top-k Jaccard, bootstrap
  intervals, and a mechanism table covering turnover, fill rate, rejected
  orders, slippage cost, and intent-to-execution gap.

Repository work:
- Promote the execution-sensitivity sweep to the main ICLR protocol. The first
  v0.3 protocol artifact is generated by `scripts/run_v03_execution_ladder.py`
  and stored under `docs/results/v0_3_execution_ladder/`.
- Keep the E2 stress-grid artifact generated by
  `scripts/run_v03_execution_stress_grid.py` under
  `docs/results/v0_3_execution_stress_grid/`; it isolates spread, latency,
  participation, impact, and combined stress axes by paired seed deltas against
  an E1 reference.
- Ensure `IntentExecutionGapEvaluator` output appears in paper tables.
- Add a generated summary that states when E3 coverage is crypto-only or
  otherwise limited.

### G3: Contamination-Control Protocol

Claim:
Financial-agent evaluation must separate no-contamination, anonymized-real, and
forward-frozen evidence.

Required evidence:
- C0 synthetic scenarios whose future paths are generated by the repository.
- C1 anonymized historical windows with symbol and timestamp masking plus
  memorization probes.
- C2 forward-frozen windows with a dated hash commitment before evaluation.
- Tables label every result by contamination tier.

Repository work:
- Freeze a v0.3 protocol spec with C0, C1, and C2 fields.
- Add a v0.3 memory-contamination artifact generated by
  `scripts/run_v03_memory_contamination.py` under
  `docs/results/v0_3_memory_contamination/`; the initial artifact implements
  C0 synthetic rows and publishes C1/C2 as control contracts.
- Add memorization-probe outputs to the results bundle.
- Prevent unlabeled real-market rows from entering the main leaderboard table.
- Keep the generated contamination-control readiness audit under
  `docs/results/v0_3_contamination_control_audit/` current so C0 fixture
  evidence is separated from C1 anonymized-real and C2 forward-frozen gaps.

### G4: Statistical Power and Claim Boundaries

Claim:
The paper distinguishes engineering, leaderboard, and scientific claims using
enough repeated evidence to support each one.

Required evidence:
- Deterministic baselines use at least 30 seeds for main tables.
- LLM rows use at least 10 seeds and 3 samples per seed for main comparisons, or
  the paper labels them as pilot evidence.
- Effect sizes, bootstrap confidence intervals, paired tests, and BH-FDR
  correction are included where model comparisons are made.
- Power curves or sensitivity notes explain which effect sizes the matrix can
  detect.

Repository work:
- Keep statistical reporting in generated artifacts, not only prose.
- Generate the v0.3 power and detectable-effect planning artifact with
  `scripts/run_v03_power_note.py`; the public bundle stores the curves and
  thresholds under `docs/results/v0_3_power_note/`.
- Generate the v0.3 variance decomposition artifact with
  `scripts/build_v03_variance_decomposition.py`; the public bundle stores
  between-seed and within-seed variance rows under
  `docs/results/v0_3_variance_decomposition/` so repeated-sample evidence is
  separated from seed/path variation before headline comparisons are written.
- Keep release checks for required statistical artifacts and columns as v0.3
  outputs expand from fixture evidence to direct API model rows.
- Keep claim text mapped to `docs/claim_boundaries.md`.
- Keep the generated v0.3 claim-boundary audit under
  `docs/results/v0_3_claim_boundary_audit/` current so public README, protocol,
  roadmap, and report-planning text do not outrun the evidence index.
- Keep `docs/results/v0_3_evidence_index/` current so pilot artifacts,
  statistical methods, and missing direct-model evidence are visible
  before paper claims are drafted.

### G5: FinAudit Trace-Auditor Track

Claim:
TreLLM can evaluate LLMs as auditors of financial-agent traces using
machine-checkable injected defects.

Required evidence:
- Defect-injection tasks cover risk violations, silent risk edits, provenance
  drift, tampered fill prices, and claim-boundary overreach.
- Scoring reports precision, recall, F1, and difficulty breakdowns.
- Self-audit and cross-audit conditions are separated to measure self-audit
  bias.

Repository work:
- Scale the existing audit-task generator from examples to a documented pilot.
  The first v0.3 FinAudit fixture pilot is generated by
  `scripts/run_v03_finaudit_pilot.py` and stored under
  `docs/results/v0_3_finaudit_pilot/`.
- Add a small direct-model pilot before the full matrix.
- Decide whether FinAudit is contribution three in the ICLR paper or a separate
  paper after the pilot effect sizes are known.

### G6: External Reproduction and Artifact Release

Claim:
The protocol can be reproduced by users who do not have access to private
workspace data.

Required evidence:
- At least three independent reproduction reports cover different environments,
  such as Windows or macOS, Linux, and Colab or Binder.
- Reproduction reports include command logs, environment metadata, manifest
  hashes, deviations, and generated result hashes.
- The public artifact bundle excludes private prompts, private caches, API keys,
  and account-specific files.

Repository work:
- Keep the external reproduction pack current with v0.3 protocol fields.
- Add a contributor-facing checklist for direct API redaction and result
  submission. The generated v0.3 checklist lives under
  `docs/results/v0_3_direct_api_submission_checklist/` and is built with
  `scripts/build_v03_direct_api_submission_checklist.py`.
- Keep release readiness checks guarding public artifact boundaries.
- Keep the v0.3 external reproduction gate generated by
  `scripts/build_v03_external_reproduction_gate.py`; reports count only when
  they are independent, schema-valid, successful, public-data-safe, and labeled
  with one required environment class.
- Track the missing independent reproduction reports in
  `docs/results/v0_3_evidence_index/v0_3_open_gaps.csv` until three external
  environments are actually present.

## Submission Kill Criteria

Do not submit an ICLR main paper if any of these remain true:

- Headline LLM comparisons still depend on Poe or another routed provider.
- Result tables do not label execution level and contamination tier.
- LLM model comparisons have too few seeds or no repeated samples.
- The main text implies trading profitability rather than evaluation reliability.
- External reproduction is only asserted by the authors.
- The public repository cannot regenerate the main protocol artifacts.

## First Implementation Milestones

Milestone 1:
Freeze `v0.3-iclr-protocol` as a spec that names the execution ladder,
contamination tiers, provider manifest fields, statistics, and evidence labels.

Milestone 2:
Run a direct-API pilot with a small model set and enough repeated samples to
exercise the manifest, statistics, and redaction path.

Milestone 3:
Regenerate the execution-sensitivity tables as main protocol evidence, including
ranking stability and intent-to-execution gap summaries. The initial C0
deterministic fixture now covers E0/E1/E2/E3, Kendall tau, top-k Jaccard, fill
rate, rejected orders, slippage cost, and intent-to-execution gap; full ICLR
headline tables still need direct API model rows and larger seed counts.

Milestone 4:
Generate a FinAudit pilot from injected defects and report both cross-audit and
self-audit performance. The initial fixture artifact now keeps the answer key
private, publishes public task hashes, and reports precision, recall, F1,
Wilson intervals, difficulty breakdowns, and a self-audit recall delta. Full
ICLR evidence still needs direct API model auditors rather than fixture
auditors.

Milestone 5:
Invite external reproduction reports against the frozen protocol and keep their
hashes in the release bundle.

## Paper Claim Template

Use this style for the main claim:

> TreLLM shows that LLM financial-agent evaluations can change under execution
> assumptions, memory contamination, and audit-trace defects. The contribution is
> a replayable evaluation protocol and evidence bundle, not a claim that a model
> can trade profitably.

Avoid this style:

> TradeArena proves that one LLM is the best trading model.
